# 다이얼로그 시스템 설계 고민

**날짜**: 2026년 4월
**카테고리**: 시스템 설계

---

## 🎯 목표

CSV 파일 기반의 유연한 다이얼로그 시스템 만들기

---

## 🤔 설계 고민 과정

### 1. 상호작용 조건 정하기

대화를 할 수 있는 조건을 먼저 정의했다.

**조건**:
- 레이어가 `Interactable`
- `Collider`를 들고 있음
- `objectData`를 들고 있음

위 3가지 조건을 만족하는 오브젝트와만 대화 가능하다.

---

### 2. CSV 파일에서 데이터를 어떻게 가져올까?

`objectData`에 `objectID`를 넣어두고, 이걸로 CSV 파일과 매칭하기로 했다.

**방식**:
1. 오브젝트의 `objectID`를 읽음
2. `.csv` 파일에서 같은 ID를 가진 줄을 찾음
3. 해당 줄을 `,`(쉼표)로 파싱해서 데이터 구분

---

### 3. CSV에 어떤 데이터가 필요할까?

대화창 UI를 먼저 떠올려봤다.

**대화창 UI 구성요소**:
- 대화창 (배경)
- 이름 박스
- Text (대사)
- portraitImg (초상화 이미지)

이걸 보고 CSV에 필요한 기본 데이터를 정리했다.

**기본 CSV 컬럼**:
- `ID`
- `Text`
- `portraitImg`

---

### 4. portraitImg 처리 - 사람 vs 사물

여기서 문제가 생겼다.

**문제 상황**:
- portraitImg는 "말을 거는 대상자(나)"가 뜨는 거임
- 근데 사물과 대화할 때는 portraitImg가 아닌 **해당 사물의 이미지**가 떠야 함

**해결 방향**:
- 사물이면 → 해당 오브젝트가 들고 있는 이미지를 띄움
- 이 이미지는 **애니메이션일 수도** 있고 **그저 특정 이미지일 수도** 있음

---

### 5. eventTrigger 시스템 설계

위 문제를 해결하기 위해 `eventTrigger`라는 부분을 따로 만들었다.

**eventTrigger 종류**:

| 명령어 | 동작 |
|--------|------|
| `ShowFrame:1` | 1번째 이미지만 띄우기 |
| `ShowFrame:4` | 4번째 이미지만 띄우기 |
| `ShowAnim` | 순차적으로 애니메이션 반복 재생 |
| `ShowOnce` | Trigger 방식으로 애니메이션 한 번 재생 후 마지막 이미지 유지 |

**ShowAnim 동작 방식**:
- 해당 오브젝트 데이터의 `FrameInterval` 값(예: 0.2초)대로 이미지를 순차 재생
- 반복 재생됨

**ShowOnce 동작 방식**:
- Trigger 방식으로 애니메이션 한 번만 재생
- 재생 후에는 마지막 이미지를 계속 유지

---

### 6. 대화 진행 방식 (nextId)

스페이스바로 다음 대화로 넘어가는 기능이 필요했다.

**방식**:
- 각 대화 데이터에 `nextId` 추가
- 스페이스바를 누르면 `nextId`를 읽어서 다음 대화로 이동
- `nextId`가 `-1`이면 대화창 종료

---

### 7. 선택지 시스템 (choiceId)

대화 중간에 선택지가 나올 수도 있다.

**방식**:
- `choiceId` 컬럼 추가
- 형식: `301|302|303` (`|`를 구분자로 사용)
- 선택지 창이 따로 뜸
- 선택지를 고르면 해당 선택지의 `nextId`로 이동

---

## 📋 1차 정리: CSV 구조

```
ID, Text, portraitImg, eventTrigger, nextId, choiceId
```

---

## 🔄 전체 흐름도

```
오브젝트 상호작용
   ↓
objectID 읽기
   ↓
CSV에서 같은 ID 찾기
   ↓
',' 로 파싱하여 데이터 추출
   ↓
대화창 UI에 표시 (Text, portraitImg, eventTrigger 실행)
   ↓
스페이스바 입력
   ↓
choiceId가 있으면 → 선택지 창 표시 → 선택한 nextId로 이동
choiceId가 없으면 → nextId로 이동
   ↓
nextId가 -1이면 → 대화 종료
```

---

## 🎮 실제 적용: 책장 상호작용

위 시스템을 책장에 적용해봤다.

**시나리오**: 책장 뒤에 숨어있는 누군가를 찾기 위해 책장과 상호작용하면서 여러 선택지를 고르는 상황

### 처음 만든 CSV

**dialogue.csv**
```csv
301,건드리기전책장,책장이다.,0,302,,ShowFrame:1
302,건드리기전책장,건드려볼까?,0,-1,301|302,ShowFrame:1
303,건드린후 책장,책이 움직였다.,0,-1,,ChangeId:305
305,건드린후책장,책이 수상한것 같다.,0,-1,,ShowFrame:4
```

**choice.csv**
```csv
301,책을 건드려 본다,303,ShowOnce
302,아니야,-1,
```

### 왜 ChangeId를 만들었나?

`ShowOnce`가 실행되면 책이 무너지는 연출이 나오고, 대화가 끝나고 다시 상호작용을 했을 때 **책이 무너져있는 장면**이 나와야 했다.

그래서 `ChangeId`를 넣어서 "다음 상호작용부터는 새로운 ID(305)로 시작하게" 만들려고 했다.

---

## ❓ 의문: 이렇게 하는 게 맞나?

선택한 것에 따라서 다음 상호작용 때 대화가 다른 게 나오게 하려고 ChangeId를 생각해냈는데, **이 방식이 최선일까?** 라는 의문이 들었다.

찾아보니 보통은 **State(상태) 형태**로 관리한다는 걸 알게 됨!

---

## 💡 개선 방안: State(상태) 시스템

### 기본 개념

**상태(State)**란 오브젝트의 "현재 모습"이다.

같은 책장이라도:
- 📚 **default** 상태 → "건드리기 전"
- 📚 **touched** 상태 → "건드린 후"
- 📚 **broken** 상태 → "부서진 후"

상태에 따라 다른 대화가 나오게 만드는 방식.

---

### 구조 변경

**1. 오브젝트 데이터에 상태 추가**

```csharp
public class ObjectData
{
    public string objectID;        // "bookshelf_01"
    public string currentState;    // "default" → "touched"로 변경됨
}
```

**2. 상태별 시작 대화 ID를 관리하는 CSV 추가**

`objectState.csv`
```csv
objectID,state,startDialogueID
bookshelf_01,default,301
bookshelf_01,touched,305
```

**3. dialogue.csv는 그대로, ChangeId 대신 SetState 사용**

```csv
ID,speakerName,text,portraitImg,nextId,choiceId,eventTrigger
301,책장,책장이다.,0,302,,ShowFrame:1
302,책장,건드려볼까?,0,-1,301|302,ShowFrame:1
303,책장,책이 움직였다.,0,-1,,SetState:touched
305,책장,책이 수상한 것 같다.,0,-1,,ShowFrame:4
```

핵심은 **`SetState:touched`** → 대화가 끝나면 오브젝트 상태를 변경

---

### 동작 흐름

**첫 번째 상호작용 (상태: default)**
```
책장에 상호작용
   ↓
현재 상태 확인 → "default"
   ↓
objectState.csv에서 시작 ID 찾기 → 301
   ↓
301 대화 시작 → 302 → 선택지 → 303
   ↓
303의 SetState:touched 실행
   ↓
책장 상태가 "touched"로 변경! 
```

**두 번째 상호작용 (상태: touched)**
```
다시 책장에 상호작용
   ↓
현재 상태 확인 → "touched"
   ↓
objectState.csv에서 시작 ID 찾기 → 305
   ↓
305 대화 시작 → "책이 수상한 것 같다."
```

---

### State 방식의 장점

**1. 확장이 쉬움**

상태를 추가하고 싶으면 그냥 추가만 하면 됨:
```csv
bookshelf_01,default,301
bookshelf_01,touched,305
bookshelf_01,broken,310       # 새로 추가
bookshelf_01,investigated,315 # 새로 추가
```

**2. 모든 오브젝트에 동일하게 적용 가능**

NPC, 보물상자, 문 등 모두 같은 시스템:
```csv
npc_01,default,401
npc_01,quest_given,410
npc_01,quest_done,420

chest_01,closed,501
chest_01,opened,505
```

**3. 저장/불러오기가 쉬움**

게임 저장 시 상태만 저장하면 됨:
```
bookshelf_01: touched
npc_01: quest_done
chest_01: opened
```

**4. 디버깅이 쉬움**

오브젝트의 currentState만 확인하면 현재 상황 파악 가능

---

## 🆚 ChangeId vs State 비교

| 항목 | ChangeId 방식 | State 방식 |
|------|--------------|-----------|
| 구현 난이도 | 쉬움 | 약간 더 복잡 |
| 상태 추가 | ID를 매번 새로 만들어야 함 | 상태만 추가 |
| 저장/불러오기 | 복잡함 | 간단함 |
| 다른 오브젝트 적용 | 매번 새로 설계 | 동일 시스템 재사용 |
| 코드 가독성 | ID 추적 어려움 | 상태로 명확 |

---

## 💡 배운 점

- 게임 오브젝트의 변화하는 상황은 **State 패턴**으로 관리하는 것이 일반적
- 직접 ID를 바꾸는 방식보다 "상태"라는 추상화 단계를 두면 관리가 훨씬 쉬워짐
- 처음부터 완벽한 설계는 어려움 → 일단 만들어보고, 문제를 발견하고, 개선하는 과정이 중요
- 작은 프로젝트면 ChangeId도 작동하지만, 확장성을 생각하면 State 방식이 좋음
